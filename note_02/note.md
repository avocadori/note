## 第1章：プロンプトエンジニアリングとは何か

### ひと言でいうと「AIへの説明書の書き方」

プロンプトエンジニアリングというと難しそうに聞こえますが、要するに **「AIに何をしてほしいか、わかりやすく説明する技術」** です。

新人スタッフに仕事を頼むときを想像してください。

- **ダメな頼み方**：「これ、いい感じにしておいて」
- **良い頼み方**：「このデータを、若手社員向けに3行で要約して。専門用語は避けて、最後に1行のまとめをつけて」

AIへの指示もまったく同じです。**指示が雑だと、雑な仕事しか返ってこない**。指示が具体的なら、人間が驚くほど質の高い仕事をしてくれます。

<!-- 📌 画像挿入位置 2：prompt_concept.png（プロンプト＝AIへの説明書のイメージ） -->

### なぜプロンプトが「AI開発の頭脳」なのか

2026年のAIツール開発では、

- **マウス操作だけでAIを動かせるツール**（Dify、Make、n8nなど。ノーコードツールと呼びます）が「体」をつくり、
- **プログラミング**（または、AIに書かせるCursorやv0）が「骨格」をつくり、
- **プロンプト**が「頭脳」をつくります。

体や骨格は数日で作れる時代になりました。差がつくのは **頭脳の質、つまりプロンプトの精度** です。

実際、企業からの受注案件で「同じツールなのに精度がまったく違う」という現象が起きるのは、99% プロンプトの差です。

### この章のまとめ

- プロンプトエンジニアリング ＝ AIへの説明書の書き方
- 2026年のAI開発は **プロンプトの質で勝負が決まる**
- 良いプロンプトを書ける人は、それだけで仕事になる

## 第2章：プロンプトの基本構造｜この型に当てはめるだけで品質が3倍

### 「神プロンプト」を分解すると、6つの要素でできている

プロのプロンプトには共通の型があります。覚えるのは6つだけ。

| 要素 | 内容 | 例 |
|------|------|-----|
| ① 役割 | AIに何者として振る舞ってほしいか | 「あなたはプロの編集者です」 |
| ② 文脈 | 前提となる状況の説明 | 「これは20代女性向けのSNS投稿です」 |
| ③ 指示 | 具体的にやってほしいこと | 「次の文章を200字に要約してください」 |
| ④ 入力 | 処理してほしい素材 | 「【文章】〜〜」 |
| ⑤ 制約 | 守ってほしいルール | 「専門用語は使わない／絵文字を3つ入れる」 |
| ⑥ 出力形式 | 答えの形を指定 | 「箇条書き3つで／表形式で」 |

### コピペで使える「基本テンプレート」

下記をベースに、6要素を埋めていけば、それだけで品質は跳ね上がります。

```
# 役割
あなたは{役割}です。

# 文脈
{前提・状況・目的}

# 指示
{やってほしいこと}

# 入力
{素材・対象}

# 制約
- {ルール1}
- {ルール2}

# 出力形式
{形・項目・長さ}
```

<!-- 📌 画像挿入位置 3：prompt_structure.png（6要素の構造図） -->

### 例：同じお願いを「型あり」「型なし」で比較

**型なしの依頼**

```
このメール、ちょっと丁寧に書き直して。
```

**型ありの依頼**

```
# 役割
あなたは大手企業の広報担当者で、ビジネスメールのプロです。

# 文脈
取引先（年上の部長）への返信メールを、社内で書いた下書きベースで丁寧にリライトしたい。

# 指示
下書きを、敬意と誠実さが伝わるビジネスメールに書き直してください。

# 入力
【下書き】
{ここに本文}

# 制約
- 過剰な敬語にならないこと（読みやすさ優先）
- 320〜400字に収めること
- 件名も提案すること

# 出力形式
件名：
本文：
```

返ってくる答えの質、明らかに違いますよね？これが **「型に当てはめる」** ことの効果です。

## 第3章：5つの基本テクニック｜知っているだけで一段階上に行ける

### テクニック①：Role Assignment（役割を与える）

AIに **「あなたは○○です」** と役割を与えるだけで、答えの質が大きく変わります。

```
あなたは経験20年のキャリアコンサルタントです。
以下の悩みに対して、3つの選択肢を提示し、それぞれのメリット・デメリットを述べてください。
```

役割を与えると、AIはその役割にふさわしい語彙・視点・深さで答えるようになります。

**コツ**：「専門家」「初心者目線の編集者」「厳しい上司」など、目的に合わせて具体的に役割を設計する。

### テクニック②：Few-shot Learning（例を見せて学ばせる）

やってほしい仕事の **「例」を2〜3個見せる** だけで、AIの出力が劇的に安定します。

```
以下のスタイルに従って、商品のキャッチコピーを5案作ってください。

【例1】
商品：ノイズキャンセリングイヤホン
→ 「世界から、雑音だけ消してくれ。」

【例2】
商品：朝専用コーヒー
→ 「目覚めたい朝に、目覚めるための一杯。」

【例3】
商品：ランニングシューズ
→ 「靴じゃない。翼だ。」

【今回の対象】
商品：{ここに商品名と特徴}
```

**コツ**：例は **3個前後** がベスト。多すぎるとAIが混乱します。

### テクニック③：Chain of Thought（手順を踏ませる）

複雑な作業は **「ステップに分けて考えてください」** とお願いするだけで、精度が跳ね上がります。

```
以下の問題を解いてください。いきなり答えを出さず、次の手順で考えてください。

ステップ1：問題の前提条件を整理する
ステップ2：必要な情報を洗い出す
ステップ3：選択肢を3つ考える
ステップ4：それぞれを比較する
ステップ5：最終的な結論を出す

【問題】
{ここに問題}
```

人間と同じで、AIも **「考えながら話す」** とミスが減るのです。

### テクニック④：Output Format Control（出力の形を指定する）

「箇条書きで」「表で」「3つに分けて」など、**形を指定する** だけで、結果がぐっと使いやすくなります。

```
次のトピックについて、以下の表形式で答えてください。

| 項目 | 内容 |
|------|------|
| メリット | |
| デメリット | |
| 注意点 | |
| 一言まとめ | |

トピック：{ここにテーマ}
```

**コツ**：表・箇条書き・見出しつき本文・JSON（プログラムが扱いやすいデータの形）など、出力の形は明確に指定する。アプリに組み込むときは特に重要。

### テクニック⑤：Constraint Specification（制約をはっきり書く）

「絶対に〜しないでください」「必ず〜してください」と **制約を明示** すると、暴走を防げます。

```
以下のルールを必ず守ってください。

- 絵文字は使わない
- 主語は「私たち」で統一
- 1段落は3行以内
- 専門用語を使ったら、その直後にかっこで言い換えを入れる
- 300字以上、400字以内
```

**コツ**：「禁止事項」より「やってほしいこと」を書く方が効きやすい。例：「ですます調を崩さない」より「すべての文末を『です』『ます』にする」。

## 第4章：そのまま使える！21個のテンプレート集

# 出力形式
- 日時／参加者：
- 議題：
- 決定事項（箇条書き）：
- TODOと担当者（表形式）：
- 次回までの宿題：
```

```

## 第5章：よくある失敗と、その直し方

### 失敗①：答えがふわっとして使えない

**原因**：制約と出力形式が指定されていない。
**直し方**：「箇条書き5個で」「表形式で」「300字以内で」など、形を必ず指定する。

### 失敗②：同じプロンプトでも、毎回結果がバラバラ

**原因**：例（Few-shot）を見せていない。
**直し方**：「こういうスタイルで」と例を2〜3個入れる。安定度が一気に上がる。

### 失敗③：AIが嘘をつく（事実と違うことを言う）

**原因**：AIは知らないことを「もっともらしく」答える性質があります。
**直し方**：「知らないことは『わかりません』と答えてください」と明示する。または、根拠を示すよう求める（例：「答えの根拠を3行で添えて」）。

### 失敗④：長い指示を出すと無視される部分が出る

**原因**：指示が長すぎてAIが優先順位をつけきれない。
**直し方**：「特に重要なルール」を最初か最後に強調する（例：「★最重要：絶対に〜しないこと」）。

### 失敗⑤：日本語が不自然になる

**原因**：英語の翻訳調になりがち。
**直し方**：「自然な日本語で書いてください」「翻訳調にしないこと」と明示する。
