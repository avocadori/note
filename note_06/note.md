## 第1章：「動くもの」と「売れる商品」の決定的な違い

### 動くもの＝技術の証明、商品＝価値の証明

「動くツール」と「売れる商品」を分けるのは、**誰の目線で作られているか** です。

| | 動くもの | 売れる商品 |
|---|----------|------------|
| 主役 | 作り手（自分） | 買い手（お客さん） |
| 評価軸 | 機能の多さ・新しさ | お客さんの困りごとを解決する深さ |
| 焦点 | 「すごい」と思われたい | 「助かる」と思ってもらいたい |
| 成功 | 完成した | 1人目が買ってくれた |
| 改善 | 機能追加 | 価値の磨き込み |

技術志向で陥りがちな罠は、**「すごいけど、誰の何を解決するのか説明できない」** 状態。これでは1円にもなりません。

### 商品化の本質は「翻訳」

商品化とは、新しいものを作ることではありません。**作ったツールを、お客さんの言葉に翻訳する作業** です。

- 「APIを使ったWebアプリ」 → 「面倒な議事録作成を5分で終わらせる相棒」
- 「自然言語でクエリ（問い合わせ命令）を投げられるダッシュボード（数値や状況をまとめた一覧画面）」 → 「Excelが苦手な現場マネージャーが、知りたいことを文章で聞ける味方」

技術の言葉を、買い手の感情に翻訳できた瞬間、**ツールは商品になります**。

### この章のまとめ

- 動くもの ≠ 売れる商品
- 商品化は「お客さん目線で翻訳する作業」
- これから渡す6つのシートが、その翻訳作業を全部支える

## 第2章：AI商品化の全体マップ｜6つのステージで考える

商品化のプロセスは、6つの段階に分かれています。それぞれに対応する **ワークシート** を、次章から順に渡していきます。

| ステージ | 問い | 対応シート |
|---------|------|-----------|
| ① 顧客解像度 | 誰の／何の／いつの悩みなのか？ | WS1 |
| ② 価値定義 | お客さんから見て、何の価値か？ | WS2 |
| ③ MVP仕様 | 最小限の機能セットは何か？ | WS3 |
| ④ プライシング | いくらで／どう課金するか？ | WS4 |
| ⑤ ローンチ＆販売 | どう知ってもらい、どう買ってもらうか？ | WS5 |
| ⑥ 継続改善 | 売れた後、どう育てるか？ | WS6 |

**順番が大事です**。①〜②を飛ばして③に飛びつくと、「機能はすごいけど誰の役にも立たない商品」が完成します。

## 第3章：ワークシート①｜顧客解像度シート

### 何のためのシートか

「お客さんが見えていない人」は、まずこの解像度を上げます。**漠然と「中小企業の経営者」** ではなく、**ピンポイントで誰か** を絞り込みます。

### 使い方

NotionやGoogleドキュメントにコピペして、空欄を埋めてください。

```
# 顧客解像度シート

## 1. 一人の具体的な人物像（ペルソナ）
- 名前（仮）：{例：田中さん}
- 年齢／性別：{例：38歳・男性}
- 職業／役職：{例：IT企業の営業マネージャー、メンバー5人}
- 1日のスケジュール：{例：8時起き、10時から商談、夕方は資料作成、19時帰宅}

## 2. 抱えている悩み（具体的に）
- 表面的な悩み：{例：日報を読む時間がない}
- 本当の悩み（背景）：{例：チームの状況を把握できず、フォローが遅れる}
- 今やっている解決策：{例：朝早く出社して全部読む}
- それでも残る不満：{例：要点が見えず、結局重要案件を見落とす}

## 3. 「いつ」その悩みを感じる？
- タイミング：{例：毎朝8時、自分の業務を始める前}
- 頻度：{例：毎日}
- ピーク：{例：四半期末は特に深刻}

## 4. お金を払ってでも解決したいと思っている度合い
- 1〜10で：{ }
- 払える金額の感覚：{例：月3,000円までなら個人で、月3万円までなら会社経費で}

## 5. その人はどこにいる？（接点）
- SNS：{X、LinkedIn など}
- 集まる場所：{セミナー、ビジネス書、コミュニティ など}
- 検索ワード：{例：「日報 効率化」「営業マネジメント ツール」}

## 6. 信頼する情報源
- 例：「特定のYouTuber」「同業の先輩」「業界メディア」など
```

### コツ

**「みんな」を想定すると「誰にも届かない」商品ができます**。一人の具体的な人物に絞ったほうが、結果的に多くの人に刺さります。

## 第4章：ワークシート②｜価値定義シート

### 何のためのシートか

「あなたの商品は、お客さんから見て **何の価値** か？」を一行で言えるようにします。

これができないと、紹介もマーケも全部ブレます。

### 使い方

```
# 価値定義シート

## 1. 30秒ピッチ（30秒で伝える短い説明）
{ターゲット}向けの、{解決する問題}を{方法}で解決する{商品タイプ}です。
{競合}と違って、{差別化ポイント}があります。

例：
「営業マネージャー向けの、日報を読む時間を1日30分→3分に短縮する、AI要約ツールです。
他の日報ツールと違って、要対応案件を自動で抽出して優先順位までつけてくれます。」

## 2. お客さんが得る価値（3つの軸）
- 機能的価値：{何ができるようになる？ 時間・お金・効率の数字で}
- 感情的価値：{どんな気持ちになる？ 安心／自信／余裕 など}
- 社会的価値：{周りからどう見られる？ 「できる人」「気が利く人」など}

## 3. ビフォー → アフター
- ビフォー：{お客さんの今の状態。困っていること、感情、行動}
- アフター：{商品を使った後の状態。何が変わるか、どう感じるか}

## 4. 「これは商品ではない」ライン
- 似ているけど違うもの：{例：単なるテキスト要約ツール、汎用AIチャット}
- なぜ違うのか：{例：営業日報という特定の文脈に特化、優先順位付けまでやる}

## 5. お客さんが言いそうな「最高の口コミ」
- 一文で：{例：「これがなくなったら、月曜の朝に戻れない」}
```

### コツ

価値定義は **1ヶ月かけても作り直す価値があります**。ここがブレたまま売り始めると、その先の全部が空回りします。

## 第5章：ワークシート③｜MVP仕様シート

### 何のためのシートか

「最小限で動く商品（MVP）」の仕様を決めます。**機能を「絞る」ためのシート** です。

機能を増やすほど商品が遠ざかります。逆です。**削るほど、商品に近づきます**。

### 使い方

```
# MVP仕様シート

## 1. 必要最低限の機能（3つだけ）
- 機能①：{例：日報を貼り付ける入力欄}
- 機能②：{例：AIによる要約と優先順位付け}
- 機能③：{例：結果を表示する画面}

## 2. やらないこと（重要！）
- やらない機能：{例：チーム管理、CRM（顧客管理システム）連携、グラフ表示、複数言語対応}
- やらない理由：{例：MVPでは検証が目的。後でいくらでも足せる}

## 3. 使う技術（軽い順に）
- AI：{ChatGPT / Claude / Gemini}
- 画面側（フロント）：{HTML/CSS/JS or Cursorで生成}
- 裏側の処理（バックエンド）：{Vercelのサーバーレス関数 or なし}
- データ保管：{ブラウザの保存機能 or Supabase（無料から使えるデータ保存サービス）}

## 4. 完成までの最短時間
- 着手から動くものまで：{ }日
- 公開できる状態まで：{ }日

## 5. テスト基準（これを満たしたらOK）
- [ ] 自分が1週間使って「便利」と感じる
- [ ] 友人や同僚3人に見せて1人以上が「使いたい」と言う
- [ ] エラーで止まる箇所がない
- [ ] 30秒以内に1回の処理が終わる

## 6. 公開URL
- {例：https://yourname.vercel.app}
- 公開予定日：{ }
```

### コツ

**「これも欲しい」「あれも入れたい」と思った時点で、MVPは失敗** に向かっています。WS3 を眺めて、「3つ以上ある機能は今やらない」と自分に言い聞かせる。これだけで成功確率が3倍。

## 第6章：ワークシート④｜プライシング設計シート

### 何のためのシートか

「いくらで売るか」「どう課金するか」を決めます。**最も多くの初心者が間違う場所** です。

### 使い方

```
# プライシング設計シート

## 1. 商品タイプ
- [ ] 単発受託（5万〜30万円／件）
- [ ] 月額継続（月1,000円〜10万円）
- [ ] ライセンス販売（一括3,000円〜）
- [ ] フリーミアム（無料＋有料プラン）

## 2. お客さんが今、解決のために払っている金額
- 同じ問題に、月いくら払っている？：{例：人件費換算で月5万円分の時間}
- 既存ツールに払っている金額：{例：Slack 月800円、Notion 月1,000円}

## 3. 価値の経済価値（数字で）
- 削減できる時間：{例：月20時間}
- 時間あたりの相場：{例：3,000円/h}
- 削減できる金額：{例：月6万円}
- → 商品の「上限価値」：{例：月6万円までは合理的}

## 4. 価格設定
- 価格：{例：月3,000円／個人、月3万円／チーム}
- 設定の根拠：{例：上限価値の1/2〜1/10}

## 5. 段階プラン（あれば）
- Free：{例：月10回まで無料}
- Standard：{例：月3,000円。回数無制限}
- Pro：{例：月3万円。チーム5人、API連携}

## 6. 値上げ・割引のルール
- 値上げのタイミング：{例：3ヶ月後、機能追加と同時}
- 早期割引：{例：最初の10人は半額の月1,500円で固定}
- 紹介割引：{例：紹介してくれた人は1ヶ月無料}

## 7. 入金フロー
- 決済手段：{例：Stripe（クレジット決済サービス）／ PayPal（オンライン決済サービス）／ 銀行振込}
- 請求書発行：{例：自動／月初／freee 連携}
```

### コツ

**価格は「自分が安いと思う2倍」で設定するくらいでちょうどいい**。多くの初心者は安く売りすぎてしまいます。お客さんが得る価値（時間削減×時給）から逆算するのが王道です。

## 第7章：ワークシート⑤｜ローンチ＆販売シート

### 何のためのシートか

「どう知ってもらい、どう買ってもらうか」を設計します。**ローンチ＝お祭り** にしましょう。地味に出すと、誰にも気づかれません。

### 使い方

```
# ローンチ＆販売シート

## 1. ローンチ日
- 公開日：{例：2026/06/01}
- 告知開始日：{例：2026/05/20（2週間前から）}

## 2. ローンチストーリー（伝える物語）
- なぜこれを作ったのか：{例：自分が毎日2時間悩んでいた}
- 開発の苦労：{例：3週間で50回プロンプトを書き直した}
- 誰の役に立つか：{例：忙しい営業マネージャー全員}

## 3. 告知チャネル（情報を流す経路。最低3つ）
- [ ] X（旧Twitter）：{例：開発ストーリーを毎日ツイート}
- [ ] note：{例：「作った経緯」記事を1本}
- [ ] LinkedIn：{例：ビジネス層向けに紹介}
- [ ] メルマガ：{例：自分のリストへ案内}
- [ ] Product Hunt：{例：海外展開も視野なら}
- [ ] 知人への直接DM：{例：3〜5人にお試しお願い}

## 4. ローンチ前1週間の発信予定
- D-7：{ティザー（予告）：「来週、新しいツールを出します」}
- D-5：{解決する問題の話：「日報レビューで困ってる人いますか？」}
- D-3：{デモ動画 or スクショ}
- D-1：{ローンチ前夜のメッセージ}
- D-0：{ローンチ告知＋URL}

## 5. 最初の10人を取る作戦
- 知人から取る：{例：DMで3人にお試し}
- SNSフォロワーから取る：{例：投稿でアンケート→興味者に個別連絡}
- 紹介で取る：{例：1人にお願いして、1人紹介してもらう}

## 6. お試し体験の設計
- 無料体験期間：{例：7日間}
- 必要な情報：{例：メールアドレスのみ}
- お試し中のフォロー：{例：3日目に「使い心地どうですか？」DM}

## 7. 「買ってもらう瞬間」の導線
- ボタン文言：{例：「7日間無料で試す」}
- ボタンの配置：{例：LP（商品紹介の1枚ページ）の上部・中央・下部の3箇所}
- 不安解消FAQ：{例：「いつでも解約できます」「メアド以外不要」}
```

### コツ

**ローンチは「点」ではなく「2週間の山」** で作る。前後の発信で空気を作ると、当日に勢いが出ます。

## 第8章：ワークシート⑥｜継続改善シート

### 何のためのシートか

ローンチは **ゴールではなくスタート** です。売れた後、商品を育てていく設計をします。

### 使い方

```
# 継続改善シート

## 1. 計測すべき数字（KPI）
- 訪問数：{例：週100人}
- 無料登録数：{例：週10人}
- 有料化率：{例：無料の20%}
- 解約率：{例：月10%以下に抑える}
- 平均利用日数：{例：週3日以上}

## 2. フィードバックを集める仕組み
- 自動アンケート：{例：解約時に1問だけ「なぜやめるか」}
- 定例インタビュー：{例：月1回、ユーザー3人と30分話す}
- 観察：{例：使われていない機能をログから特定}

## 3. 改善の優先順位
- 緊急かつ重要（即修正）：{例：エラー、決済不具合}
- 重要だが急がない（次回リリース）：{例：UI改善、機能追加}
- 急ぐが重要でない（後回し）：{例：些細なテキスト修正}
- 急がず重要でない（やらない）：{例：マニアックな要望}

## 4. リリースサイクル
- 大きなアップデート：{例：月1回}
- 小さな改善：{例：随時}
- バグ修正：{例：発見次第}

## 5. 価格・プラン見直しのタイミング
- 3ヶ月後の見直し：{ユーザー数／満足度／継続率を見て}
- 値上げの目安：{機能追加・ユーザー満足度が高いとき}

## 6. お客さんとの関係づくり
- 配信：{月1ニュースレターで「新機能」「Tips」「事例」を共有}
- コミュニティ：{Discord / Slack / オープンチャット}
- 個別フォロー：{大口顧客には個別DM}

## 7. 拡張プラン（次の半年）
- 機能拡張：{例：チーム機能、APIエクスポート}
- 顧客層拡張：{例：個人 → チーム → 企業}
- 横展開：{例：営業日報 → 採用面談 → 顧客サポート分析}
```

### コツ

**売れたツールは、磨くほど育ちます**。3ヶ月で売れたら、次の3ヶ月でユーザーを倍に。半年で同じテンプレを別業種にも横展開。これが10万円→30万円→100万円への道です。

## 第9章：90日商品化ロードマップ

ここまでのワークシートを、実際の時系列に落とし込みます。

### 月1（1〜30日）：解像度を上げる

| 週 | やること | 完了基準 |
|---|---|---|
| 1週目 | WS1（顧客解像度）を埋める | 一人の人物像が具体的に描ける |
| 2週目 | WS2（価値定義）を埋める | 30秒ピッチが書ける |
| 3週目 | 想定ターゲットに5人ヒアリング | 仮説の修正点が見つかる |
| 4週目 | WS1／WS2を改訂 | 「これでいける」と確信できる |

### 月2（31〜60日）：作って試す

| 週 | やること | 完了基準 |
|---|---|---|
| 5週目 | WS3（MVP仕様）を埋める | 機能が3つに絞れる |

## 第10章：商品化でよくある失敗

### 失敗①：機能の足し算で永遠にローンチできない

「もう少し機能を……」と続けていると、3ヶ月たってもLPすら作っていない、という状態に。

**回避策**：WS3 の「3機能ルール」を死守。日付を先に決める。

### 失敗②：価格を安くしすぎる

「最初は安く」と思って500円にしたら、サポートコストで赤字。

**回避策**：WS4 の「上限価値」から逆算。最低でも価値の1/10。

### 失敗③：ローンチを地味にやる

ひっそり公開すると、ひっそり誰にも見つかりません。

**回避策**：WS5 の「2週間のお祭り」設計を必ず実行。

### 失敗④：1人目の有料顧客を取れずに諦める

**回避策**：「無料体験中の3日目DM」を必ず入れる。直接「どうですか？課題ありますか？」と聞く。

### 失敗⑤：継続改善をやらず、買い切り型のまま放置

「売れた！」で満足して放置。3ヶ月後にはユーザーがゼロに。

**回避策**：WS6 で月次の改善サイクルを必ず回す。
